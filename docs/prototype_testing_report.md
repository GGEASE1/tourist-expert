- отдельно вызывал:

- `selected_rule`;
- `matched_rules`;
- `recommendation`;
- `elapsed_ms`;

То есть у обратного вывода уже была развитая линейная цепочка доказательства и дерево доказательства.

### Как считается число шагов и проходов для прямого вывода

- число шагов = `len(result.steps)`;
- число проходов = `result.passes`.

В текущей учебной модели проходы интерпретируются так:

1. ввод исходных фактов;
2. полный проход по правилам;
3. разрешение конфликта и выбор результата.

## 6. Как теперь реализованы 2 примера обратного вывода

Для тех же двух сценариев выполняется обратный вывод с целью `*`.

Почему это сделано именно так:

- цель `*` означает: доказать, **какая рекомендация из всей базы знаний** подходит данному пользователю;
- это честный backward-сценарий;
- он даёт длинную полноценную трассировку с проверкой многих правил, подцелей, отказов и подтверждений;
- поэтому требование **не менее 51 шага** выполняется без искусственного подрисовывания.

### Обратный пример 1

- цель: `*`
- подтверждённая гипотеза: `warm-relax-premium`
- число шагов: **569**
- число проходов: **3**

### Обратный пример 2

- цель: `*`
- подтверждённая гипотеза: `winter-active-ski`
- число шагов: **567**
- число проходов: **3**

### Что видно в цепочке обратного вывода

На странице теперь пошагово отображается:

- постановка цели;
- выбор правил-кандидатов;
- попытка доказать каждое правило;
- формирование подцелей по условиям;
- подтверждение подцели фактами;
- отказ по неподтверждённым условиям;
- доказательство правила;
- выбор итоговой гипотезы.

### Как считается число шагов и проходов для обратного вывода

- число шагов = `len(result.steps)`;
- число проходов = `result.passes`.

В текущей трассировке backward-проходы соответствуют уровням доказательства:

1. цель;
2. правила-кандидаты;
3. подцели по фактам.

## 7. Как измеряется время выполнения

Измерение времени уже было в движке и сохранено:

- для прямого вывода используется `perf_counter()` внутри `evaluate(...)`;
- для обратного вывода используется `perf_counter()` внутри `backward(...)`.

На `/test` теперь показывается:

- время выполнения каждого примера прямого вывода;
- время выполнения каждого примера обратного вывода;
- среднее время прямого вывода;
- среднее время обратного вывода;
- краткое текстовое сравнение.

### Контрольный локальный замер на 5 апреля 2026

Так как время вычисляется заново при каждом запуске страницы, численные значения
слегка меняются. Поэтому в отчёте важно фиксировать не “абсолютно неизменные”
миллисекунды, а сам факт измерения и порядок величин.

На одном из контрольных запусков средние значения были такими:

- среднее время прямого вывода: `2.041 ms`
- среднее время обратного вывода: `0.839 ms`

Именно поэтому на странице `/test` время показывается динамически для каждого
запуска, а в блоке производительности дополнительно выводится краткое текстовое сравнение.
